# Plan — Connections

Wire protocols between every pair of components. Use this as the checklist before writing any new call.

> Source of truth: this file. If [system-map.md](../system-map.md) disagrees, fix both in the same change.

## 1. Client ↔ Hub

### REST

- TLS 1.3, HTTP/2
- Cookies: `access_token` (httpOnly, secure, SameSite=Lax, 15min), `refresh_token` (httpOnly, secure, SameSite=Strict, 30d)
- CSRF: double-submit token for mutating requests
- JSON bodies; `application/json`
- Standard error shape (see [api-contracts.md](../api-contracts.md#error-shape-all-apis))
- Rate limits: 60/min/IP, 600/min/user

### WebSocket

- `wss://hub/api/v1/ws/...`
- Auth via query param `?token=<access_token>` (short-lived) OR via initial JSON frame after upgrade
- Heartbeat: ping every 30s; server disconnects on 60s silence
- Subscription multiplexing: single socket → multiple channels by JSON `{type:"subscribe", channel:"agent/{id}/status"}`
- Reconnect: exponential backoff 500ms → 16s max; client re-subscribes on reconnect

### Cold-wake response

- If user VM is stopped, hub returns `202 {error:{code:"VM_WAKING"}}` with `Retry-After: 3`
- Client shows shimmer, retries 500ms/1s/2s/4s/8s for ≤15s total
- Successful wake before timeout → normal response

## 2. Hub ↔ User-Server (mTLS)

### Cert pipeline

- Hub acts as CA; issues per-VM cert at provision
- Certs stored on VM at `/etc/user-server/tls/{cert.pem,key.pem}`
- Rotated on each wake (new cert minted, pushed via `/hub/secrets/refresh`)
- Expiry: 24h; rotation on wake always fresh

### Hub → User-Server (client side in hub, server in user-server)

Base URL: `https://<vm-internal-endpoint>:8443`. TLS via hub's client cert; server validated against hub CA.

Request headers:
- `X-Hub-Secret: <per-VM shared secret>`
- `X-Request-ID: <uuid>`
- Standard `Content-Type`

All `/api/v1/u/*` client calls are forwarded with path rewritten: `/api/v1/u/agents/{id}` → `/agents/{id}`. Method, body, query string preserved. Headers stripped of client auth; `X-User-ID` and `X-User-Email` added for audit.

WebSocket proxy: hub upgrades client connection, opens downstream WSS to user-server, pipes frames both ways. On user-server disconnect → hub closes client with code 1011.

### User-Server → Hub (user-server client, hub server)

Base URL: `https://hub-internal:9001`. TLS via user-server client cert; hub server cert from platform CA.

Endpoints used:
- `POST /internal/v1/indexer/flush`
- `POST /internal/v1/credits/heartbeat` (reconciliation; hub is authority)
- `POST /internal/v1/otel/spans`

Request headers:
- `X-UserServer-Token: <rotated per wake>`
- `X-VM-ID: <provider_vm_id>`

### Drain + halt control plane

- `POST /hub/drain` — graceful shutdown, waits ≤30s
- `POST /hub/halt` — immediate stop of all agents (credits depleted)
- `POST /hub/secrets/refresh` — push decrypted user secrets post-wake
- `GET /hub/indexer/since?ts=...` — indexer pull sync

All are **hub-initiated**, idempotent, return 200 with small JSON ack.

## 3. User-Server ↔ Agent (VM-internal)

### Network

- Docker internal bridge `artic-net` inside the VM
- Agent container addressable by `artic-agent-{id}:8000`
- No public exposure; not even on VM public IP

### Auth

- `X-Internal-Secret: <per-spawn uuid>`
- Value generated by user-server at spawn; injected as agent env
- Rotated every VM wake

### User-Server → Agent

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/health` | spawn readiness |
| POST | `/start` | begin tick loop |
| POST | `/stop` | graceful stop |
| GET | `/status` | current state |
| GET | `/logs?limit=N` | ring-buffer tail |

### Agent → User-Server (push)

| Method | Path | Frequency | Body |
|--------|------|-----------|------|
| POST | `/agents/{id}/status` | every tick | price, position, pnl |
| POST | `/trades` | on open/close | trade record |
| POST | `/logs` | every 10 ticks | `[LogEntry]` |
| POST | `/supervisor` | on supervisor decision | action, reasoning |
| POST | `/llm/plan` | on plan request | context |
| POST | `/llm/supervise` | supervisor tick | context |
| POST | `/llm/chat` | chat endpoint | messages |

Timeouts: 5s connect, 10s read. On failure: agent logs `warn`, keeps running; state will reconcile on next tick.

## 4. User-Server → LLM Provider

- HTTPS to provider endpoint (openai.com, anthropic.com, googleapis.com, api.deepseek.com)
- Auth via user's API key cached in user-server memory
- Retry: 2 attempts on 5xx/429; no retry on 4xx
- Timeout: 20s
- Rate limit per user: 60/min across all providers
- Never logged in plaintext — only token counts + latency

## 5. User-Server → Chain RPC

- HTTPS JSON-RPC
- Wallet key in memory (decrypted at boot from VM keystore)
- Sign tx locally with `eth_account`, send `eth_sendRawTransaction`
- Poll `eth_getTransactionReceipt` until mined (max 120s, retry with +20% gas on timeout, max 3 rounds)
- On success: write `indexer_tx` row
- On final fail: log WARN, emit metric, do not retry

## 6. Agent → Pyth Hermes

- Public HTTPS, no key
- Endpoint: `https://hermes.pyth.network/api/latest_price_feeds?ids[]=<id>`
- Timeout: 2s; on fail skip tick
- Feed IDs resolved dynamically from Pyth symbol catalog at agent start (not hardcoded)

## 7. Hub → Market Provider

### TwelveData
- HTTPS; `apikey` query param
- Rate budget: 8/min (platform-wide; hub scheduler enforces)
- Response cached 60s in `market_cache`

### Pyth Hermes (hub-side for WS broadcast)
- SSE or WebSocket to `hermes.pyth.network`
- Fan out via hub `/ws/prices`

## 8. Hub → VM Provider

- Provider-specific SDK (Fly / Firecracker host)
- Operations: `create_machine`, `start`, `stop`, `snapshot`, `resume`, `destroy`
- Auth: provider API token in hub env
- On provider outage: hub degrades — blocks new signups, returns 503 on wake, keeps running VMs alive

## 9. Hub → Chain RPC (funder)

- Platform hot wallet in hub env (testnet)
- Signs + sends via RPC
- Same retry policy as user-server → chain

## Invariants across all connections

- Every hop authenticates; no "trusted network" shortcuts
- All tokens/secrets rotated on VM wake
- Timeouts on every call — default 10s if not specified
- Retries only on network/5xx/429; never on 4xx or auth errors
- All cross-component calls carry `X-Request-ID` for trace correlation
- OTel span per outbound call with duration, status, and request ID
